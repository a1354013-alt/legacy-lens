# Legacy Lens

Legacy Lens imports a legacy codebase, runs a persisted structural analysis workflow, and exports a ZIP report generated from the same saved analysis result shown in the UI.

## Supported product scope

- Create a project
- Import source from ZIP or Git
- Persist extracted files to MySQL
- Run server-owned analysis and persist the result
- Inspect the saved result in the UI
- Download a ZIP report generated from the persisted analysis row
- Delete the full project graph

## Supported languages

Legacy Lens currently supports import and analysis only for:

- Go (`.go`)
- SQL (`.sql`)
- Delphi (`.pas`, `.dpr`, `.delphi`, `.dfm`, `.inc`, `.dpk`, `.fmx`)

Delphi form and package files such as `.dfm`, `.inc`, `.dpk`, and `.fmx` are imported and recognized, but they may be analyzed with reduced heuristics compared to `.pas`/`.dpr` source files. Unsupported code files are skipped during import with explicit warnings. They are not treated as analyzed source.

## Analysis output

- One `analysisResults` row per project
- Symbols
- Symbol dependencies
- Fields
- Field dependencies
- Risks
- Derived rules
- Markdown and YAML report documents

## Accuracy and limitations

- The analyzer is heuristic, not compiler-grade semantic analysis.
- Go, SQL, and Delphi symbol/dependency extraction can miss, degrade, or skip ambiguous cases by design.
- Multi-line SQL, dynamic SQL, and Delphi cross-unit resolution are best-effort only.
- Warnings in the UI and ZIP summary should be reviewed before using the result as source-of-truth.

## Security notes

- OAuth login uses a server-signed state token with nonce, timestamp, redirect intent, and validation on callback.
- Session token creation and verification share one schema.

## Data consistency strategy

- The database schema enforces one `analysisResults` row per project.
- Drizzle schema and migrations define foreign keys and cascade relationships for project-owned data.
- Service-layer transactions remain the primary safeguard for import and analysis write ordering.

## Repository hygiene

Release source should contain source files, migrations, tests, and docs only. It should not include:

- `.git`
- `node_modules`
- `dist`
- `.manus-logs`

`dist/` is a build artifact generated by `pnpm build`, not a source-of-truth directory.

## Local setup

```bash
corepack pnpm install
corepack pnpm db:migrate
corepack pnpm dev
```

Default local URL:

- `http://localhost:3000`

## Scripts

- `corepack pnpm dev`: start the development server
- `corepack pnpm build`: build client and server bundles
- `corepack pnpm start`: run the production bundle
- `corepack pnpm typecheck`: type-check application code
- `corepack pnpm typecheck:test`: type-check test files
- `corepack pnpm check`: run both type-check passes
- `corepack pnpm lint`: run ESLint
- `corepack pnpm test`: run Vitest
- `corepack pnpm test:watch`: run Vitest in watch mode
- `corepack pnpm db:generate`: generate Drizzle migrations from schema changes
- `corepack pnpm db:migrate`: apply Drizzle migrations
- `corepack pnpm db:push`: generate then apply migrations

## Workflow states

Project lifecycle:

- `draft`
- `importing`
- `ready`
- `analyzing`
- `completed`
- `failed`

Analysis result lifecycle:

- `pending`
- `processing`
- `completed`
- `partial`
- `failed`
